Prompt Engineering

How to speak to a mind that doesn't work like yours — and reliably get what you want.

"Prompt engineering is the art of speaking a language that doesn't exist anywhere in nature, to a mind that doesn't exist anywhere in nature, to get results that used to require a team."

The Intuition

Prompts are soft programs

A prompt isn't a search query — it's a program. Not written in Python or C, but in the same statistical fabric the model was trained on. When you write a prompt, you're not asking a question. You're specifying a distribution over outputs. The model samples from that distribution. Your job is to make the target distribution narrow enough to be useful.

This reframe matters immediately. A bad prompt doesn't "confuse" the model — it leaves the distribution too wide. A vague instruction like "write a summary" allows thousands of valid completions: bullet points, prose, one sentence, five pages. A precise instruction — "write a 3-sentence summary in plain English for a non-technical reader, starting with the main conclusion" — narrows the distribution to a tight cluster. Same model, same weights, radically different output.

  • Tokens, not words: the model sees a sequence of subword tokens. "Unhelpfulness" might be 3 tokens. Token boundaries affect how the model computes context — which is why splitting ideas across lines sometimes helps.
  • Context window as working memory: everything in the prompt is "active" at inference time. The model has no persistent memory beyond this window. What you include is the entirety of what it knows about your task.
  • Attention as routing: each token attends to every other. Putting important instructions early and repeating them at the end statistically increases the attention weight they receive — this isn't a trick, it's how transformers work.
  • Completion bias: the model is trained to predict the next token. It doesn't have goals — it has completion instincts. A well-structured prompt exploits that instinct, setting up a "completion trajectory" that naturally leads to the output you want.

Understanding It Deeply

Build it from first principles — don't just accept it

Chain-of-thought, few-shot ordering, zero-shot drift — these aren't magic tricks. They each have mechanistic explanations grounded in how transformer attention and token prediction work.

Think first Why does chain-of-thought prompting improve reasoning — what is happening inside the model when you add "let's think step by step" versus asking for the answer directly?

The model doesn't "reason" in the human sense — it predicts the next token. When you ask for the answer directly, the model must compress all intermediate logic into the next token distribution, which it fails at for complex problems. Chain-of-thought turns reasoning into a generation problem: by producing intermediate tokens, the model creates a richer context that makes subsequent correct tokens more probable. Each reasoning step is simultaneously output and new context. You're not getting a smarter model — you're getting a model that uses its own output as scaffolding.

Think first What is the difference between zero-shot, one-shot, and few-shot prompting at a mechanistic level? Why does showing examples work at all — the model's weights don't change?

Few-shot prompting works through in-context learning — a phenomenon where the transformer's attention mechanism uses the examples to implicitly condition the output distribution without any gradient updates. The examples create a "task frame" in the context window: the model infers the pattern (format, tone, logic, output structure) by attending to example pairs and extrapolates that pattern to the new query. This is gradient descent without parameter updates — the examples steer attention heads at inference time. Zero-shot works only when the task pattern is well-represented in the training distribution. One-shot and few-shot work better precisely when it isn't.

Think first Why does the order of examples in a few-shot prompt matter? If the examples carry the same information regardless of order, why would order affect the output?

Two effects are at work. First, recency bias: the final few tokens in a long context receive disproportionate attention due to how causal attention masks are structured — later examples are "closer" to the prediction site. This means the last example disproportionately shapes the output format and tone. Second, label distribution bias: if you show 3 positive examples followed by 1 negative, the model's default completion skews positive, regardless of the test input. Research by Zhao et al. (2021) showed that order alone can swing accuracy by 30+ percentage points on classification tasks. The fix: calibrate against a neutral prefix, or balance examples deliberately.

Eight techniques — when to use each one

From Theory to Production

How real systems use prompt engineering

Production prompt engineering is not "write better prompts." It is a systematic engineering practice with version control, evaluation harnesses, and documented failure modes. Here's what it actually looks like.

Claude's system prompt architecture. Anthropic publishes the structure of Claude's prompt layers: a core system prompt (sets values and core behaviors), an operator system prompt (business-specific persona and constraints), and user turns. Each layer can expand but not override safety constraints from the layer above. This is prompt engineering at infrastructure scale — Claude's "personality" on Claude.ai versus its personality in an API deployment are both controlled by this layered prompt architecture, not by different model weights.

OpenAI's prompt engineering guide (published 2023) identifies six principal strategies: write clear instructions; provide reference text; split complex tasks into subtasks; give the model time to think (chain-of-thought); use external tools; and test systematically. The guide emphasizes that prompt changes should always be evaluated against a held-out test set — not judged by feel on three examples. This mirrors software engineering discipline: changes require tests.

Customer support at scale. A tier-1 support system at a large e-commerce company might use: (1) a system prompt that sets agent persona, escalation criteria, and tone guidelines; (2) retrieved context injected via RAG containing the user's order history; (3) few-shot examples of correctly-handled edge cases; (4) an output schema forcing structured handoff fields (resolution_type, escalate: bool, draft_reply: string). The human support agent sees only the draft_reply — the structured fields feed directly into the CRM. The prompt is a machine interface, not a conversation.

Code generation. GitHub Copilot and Claude Code both use context-sensitive prompts that include: the file being edited, recent diff history, open tabs for cross-file context, and a repo-level system prompt configured by the developer. The model never sees "write code" — it sees a rich structured context that makes the next token completion naturally be useful code. Every design decision follows from token prediction, not magic.

The consistent pattern: Every production use case structures prompts in the same way — system context first, task-specific instructions next, output format last. The model isn't being asked to figure out what you want. It's being given a completion trajectory to follow. The art is making that trajectory inevitable.

Common Misconceptions

Five causal misconceptions — and the real mechanism

These aren't just wrong beliefs. Each one misidentifies the cause of a real observation, which leads to engineering decisions that make things worse.

  • ① More instructions = better output.
    Adding instructions increases specificity only if the instructions are unambiguous and non-contradictory. A long prompt full of vague qualifiers ("be helpful but not too formal, professional but approachable") doesn't narrow the output distribution — it widens it by introducing conflicting constraints. The model averages over the conflicts. Precise, minimal, non-overlapping instructions outperform verbose ones.
  • ② Temperature is always the answer to bad outputs.
    Temperature controls the sharpness of the sampling distribution — it does not change the center. If the peak of the distribution is the wrong output, lowering temperature makes you get that wrong output more consistently. Raising it adds noise but doesn't inject correctness. Temperature is a correctness amplifier, not a problem solver. Fix the prompt first; tune temperature last.
  • ③ A prompt that works on GPT-4 will work on Claude (or vice versa).
    Prompts encode assumptions about the model's instruction-following style, tokenization, and training distribution. Different models have been trained with different RLHF pipelines, system prompt architectures, and safety tuning. A prompt that relies on GPT-4's formatting conventions may elicit a completely different behavior from Claude, Gemini, or Llama. Prompts are model-specific artifacts. Always evaluate on the target model.
  • ④ Prompts are stable across time — the same prompt should give the same output.
    Model providers continuously update their models without changing version labels. OpenAI's "gpt-4" endpoint has served at least a dozen different model snapshots. Claude's behavior changes with training updates. A prompt that worked reliably in January may degrade by March due to a behind-the-scenes model swap. Production systems must pin to specific model versions and run regression evaluations on updates.
  • ⑤ The model "understands" your intent — it can infer what you mean.
    The model predicts the next token given the context. It has no persistent goals, no theory of mind about you, and no access to your unstated intent. What looks like "understanding intent" is statistical pattern-matching against training data that happened to cover similar prompts. When the model "gets it," it's because your prompt is close enough to patterns it learned. When it doesn't, there's no appeal — you must make the intent explicit in the tokens.

Summary

Six things to lock in

  • A prompt specifies a distribution over outputs, not a command. Precision narrows the distribution; vagueness widens it. Your job is to make the target completion statistically inevitable.
  • Chain-of-thought works because generating intermediate tokens creates richer context that makes subsequent correct tokens more probable. It turns reasoning into a generation problem.
  • Few-shot prompting works through in-context learning — the attention mechanism infers task patterns from examples without updating any weights. Example order and label balance both bias the output.
  • Production prompts have layers: system context → task instructions → output schema. Each layer does a different job. Mixing them makes both jobs harder.
  • Prompts are model-specific, version-sensitive artifacts. They encode assumptions about training, tokenization, and instruction-following style. Always evaluate on the target model, pin to specific versions.
  • Temperature does not fix a bad prompt. Fix the prompt; tune temperature last. The model has no access to your unstated intent — make everything explicit in tokens.

Test Your Understanding

9 Questions

0 / 9 revealed

01 You add "let's think step by step" to a math problem and accuracy improves dramatically. A colleague says the model is now "thinking more carefully." What is actually happening? ▼
The model is generating intermediate tokens that enrich the context for subsequent predictions. It isn't "thinking more carefully" — it's predicting the next token, just with more relevant preceding context. Each reasoning step appears as output, then becomes context for the next step. The model that couldn't produce the right answer in one token can produce it across many tokens, because each token conditions the next. It's a generation scaffold, not a cognitive change.
02 Your few-shot prompt has 5 examples: 4 positive sentiment, 1 negative. The test input is clearly negative but the model labels it positive. What is the most likely cause? ▼
Label distribution bias. Zhao et al. (2021) showed that the label distribution of few-shot examples biases the model's output distribution toward the majority label, independent of the test input's actual content. With 4 positive and 1 negative example, the model sees a 4:1 ratio and anchors toward "positive." Fix: balance examples equally across labels, or use a calibration prompt that normalizes against a content-free baseline.
03 You have a prompt that works perfectly on GPT-4. You switch to Claude and get different, worse results. Your team says "Claude is a worse model for this task." What's the more likely explanation? ▼
The prompt encodes model-specific assumptions. Different models have different instruction-following conventions, system prompt architectures, and RLHF tuning. A prompt that relies on GPT-4's behavior on certain cues (like specific XML tags, role framing, or output format instructions) may elicit different behavior from Claude. The prompt is a model-specific artifact, not a universal specification. The fix is to evaluate and tune the prompt on the target model — not to conclude the model is inferior.
04 Your model keeps returning prose when you need JSON. A colleague suggests raising the temperature to "give it more freedom." Will this help? ▼
No — temperature doesn't fix a misdirected output distribution. Temperature sharpens or broadens the sampling distribution around the existing peak. If the peak is "prose," higher temperature adds noise; lower temperature makes prose more deterministic. Neither moves the peak to "JSON." The fix is in the prompt: add an explicit output format instruction ("Respond ONLY with valid JSON"), show a JSON example, or use a structured output API parameter if available. Fix the prompt first; tune temperature last.
05 You're building a customer support bot. What order should you put the prompt components in, and why? ▼
System context → task instructions → output schema. System context (persona, constraints, tone) goes first because it establishes the frame for all subsequent tokens. Task-specific instructions come next to define the current operation. Output schema goes last because it is the completion target — the model's prediction should flow toward it. This order also reflects attention dynamics: early tokens set persistent context; the final specification anchors the completion. Mixing these layers (e.g., schema before context) creates ambiguity in how constraints interact.
06 A prompt worked reliably for 6 months. Then it starts producing worse outputs — you changed nothing. What's the most likely cause? ▼
The underlying model was silently updated. Model providers regularly update models behind stable endpoint names. OpenAI's "gpt-4" and "gpt-4-turbo" endpoints have served multiple distinct model versions. Your prompt was tuned against a previous snapshot; the new snapshot behaves differently. The fix is to pin to specific versioned model identifiers (e.g., "gpt-4-0613") and run a regression test suite before any version migration. Production LLM systems require the same engineering discipline as any software dependency.
07 When would you use self-consistency prompting over a single chain-of-thought pass? ▼
When the cost of a single wrong reasoning chain is high and compute budget allows. Self-consistency generates N independent chains of thought, then majority-votes the final answer. A single CoT chain can fail if it takes a wrong reasoning path early. Self-consistency hedges against this by sampling multiple paths — diverse chains that reach the same answer are more likely to be correct. Use it for high-stakes single-answer problems (math, medical reasoning, legal analysis). Don't use it for tasks where you need one structured output, or where latency/cost matters — it runs the model N times.
08 You have a 10,000-token prompt with important instructions in the middle. The model seems to ignore them. Why, and how do you fix it? ▼
Lost-in-the-middle effect. Research shows that transformer attention tends to under-weight tokens in the middle of very long contexts — priority goes to the beginning (for overall framing) and the end (for recency/completion proximity). Instructions buried in the middle of a long document-retrieval prompt are statistically less attended to. Fixes: (1) move critical instructions to the beginning and repeat them at the end; (2) use XML tags to create structural anchors; (3) reduce prompt length by summarizing context rather than pasting it raw; (4) use models explicitly trained for long-context instruction following.
09 What is the difference between a system prompt and a user prompt, and why does it matter for security? ▼
System prompts establish operator-level constraints; user prompts are runtime input. In Anthropic's architecture: operator system prompts are set by the business deploying the model and carry higher trust — they define persona, allowed topics, and safety constraints. User prompts are runtime input and carry lower trust — the model is trained to be helpful but not to override system-level constraints based on user requests. This matters for security because prompt injection attacks attempt to override system constraints through user input ("ignore previous instructions and..."). A well-designed system treats the distinction as a trust boundary, not just an ordering convention. System prompt contents should be treated as sensitive configuration, not user-visible state.

Part 1 · Basic Machine Learning

Decision Trees & Random Forests

The algorithm behind every flowchart in medicine, finance, and fraud detection.

Every diagnostic decision tree a doctor uses — "fever above 38°C? → check for rash → rule out meningitis" — is this algorithm. The same recursive splitting logic that triage nurses follow on laminated cards is the core of one of the most widely deployed ML models in the world. You already understand the intuition. This chapter makes the math explicit.

The Intuition

The 20-questions game — played recursively

A decision tree learns to classify data by asking a sequence of yes/no questions about features, splitting examples into smaller and purer groups at each step — exactly like the game of 20 questions, but the machine picks the best question at each turn.

Think of it this way: you have a bag of mixed fruit and want to sort them. Your first question might be "is it longer than 10 cm?" — that separates bananas from most others. Then "is it yellow?" splits the remaining. Each question divides the current group in two. Recursion means the process repeats independently inside each resulting group until every subgroup is pure (all one class) or small enough to stop.

  • The root node is the first question — asked of every training example.
  • Each internal node tests one feature against a threshold: X₁ ≤ 3.5?
  • The branches are the yes/no outcomes of each test.
  • The leaves are the final answers — the predicted class (or value).
  • The algorithm is greedy — it picks the locally best split at each step, not the globally optimal tree.

This is recursive binary partitioning: the input space gets sliced into rectangular regions by axis-aligned cuts, and each region votes for one class. What you see in the animation next is literally this geometry unfolding.

Understanding It Deeply

Build the algorithm from first principles

Three questions get to the core of how decision trees actually work — and why random forests exist. Pause and think through each one before reading the answer.

Think first The tree has to choose which feature to split on at each node. It could split on age, income, or credit score. How should it decide? What property should the chosen split maximize or minimize?

It minimizes impurity — a measure of how mixed the classes are in the resulting child nodes. The best split is the one that, after dividing the data, produces the two purest groups. A perfectly pure node contains only one class; the most impure node is a 50/50 mix. The algorithm evaluates every feature and every possible threshold for that feature, computes the weighted impurity of the resulting split G(Q, θ) = (n_left/n) · H(left) + (n_right/n) · H(right), and picks the θ* that minimizes G. The most common impurity measure is Gini impurity — see The Math section for the formula.

Think first Imagine letting the tree grow until every leaf contains exactly one training sample. Training accuracy would be 100%. What happens on the test set? And what structural property of the tree explains why?

A fully-grown tree memorizes the training data rather than learning the underlying pattern. With one sample per leaf, the tree has essentially stored every training example as a lookup rule — including noise, label errors, and random fluctuations. On new data, these rules fail because they were never patterns to begin with. This is overfitting. Scikit-learn's docs make the mechanism precise: each internal node requires a minimum of min_samples_split samples; each leaf requires min_samples_leaf. Setting max_depth is the bluntest and most common regularization — cap depth at 3–5 and the tree generalizes far better. Cost-complexity pruning (ccp_alpha) offers a principled alternative: it penalizes tree size directly, removing the weakest branches until the size-accuracy tradeoff is optimal.

Think first A single pruned decision tree has reasonable accuracy. Training 100 trees on the same dataset produces 100 near-identical trees that vote identically — so accuracy doesn't improve. What would you have to change about how the trees are built to make an ensemble actually better than any individual member?

You need the trees to be diverse — making different errors so they cancel out when averaged. A random forest achieves this two ways. First, each tree trains on a different bootstrap sample (sampling with replacement) — so each tree sees a different version of the data. Second, at each split, only a random subset of features is considered (typically √p for classification) — so even when the same data point reaches two different trees, they may split it on different features. These two injections of randomness decorrelate the trees. When their diverse errors are averaged, variance falls dramatically while bias barely increases — the core insight of bagging (bootstrap aggregating).

The Math

Gini impurity and how splits are scored

Every split decision reduces to one calculation: which cut produces the lowest weighted impurity in the two child nodes? Gini impurity quantifies how mixed a node's classes are.

Gini Impurity
Gini(m) = Σ_k p_mk · (1 − p_mk) = 1 − Σ_k p_mk²
m the current node being evaluated k class index (e.g., k = 0 for negative, k = 1 for positive) p_mk proportion of class k examples in node m Gini = 0 perfectly pure node — all one class Gini = 0.5maximally impure — 50/50 split between two classes

Why this formula? Think of randomly picking two samples from a node and asking if they're different classes. The probability they disagree is exactly the Gini score. A pure node — all one class — means every pair agrees, so Gini = 0. A 50/50 node maximizes disagreements, giving Gini = 0.5. We want nodes where all samples agree — so we minimize Gini.

Four-step worked example — splitting 8 samples on feature X₁:

SetupNodeClassesGini
1 Parent node (all 8) n = 8 5 blue, 3 red 0.469
2 Try X₁ ≤ 3.0 left: n=5 4 blue, 1 red 0.320
3 Right child right: n=3 1 blue, 2 red 0.444
4 Weighted Gini G(θ) 5/8×0.32 + 3/8×0.44 0.367
Improvement: 0.469 → 0.367. The algorithm accepts this split. Now the same process repeats inside each child node, recursively. The split with the best impurity reduction across all features and thresholds is always chosen. This greedy strategy can't guarantee a globally optimal tree — but it's O(n·p·log n) per level, making it tractable.

Entropy vs Gini? Entropy (−Σ p_mk · log(p_mk)) is the alternative criterion. It's theoretically motivated by information theory and penalizes impurity slightly more aggressively at moderate mixes. In practice, both criteria produce similar trees — the difference rarely justifies the extra computation of the log.

From Toy to Real

Medical diagnosis to credit scoring to high-dimensional reality

The 2D animation shows the algorithm's geometry cleanly. Here's what it looks like when applied to problems that actually matter — and what changes when you move from 2 features to 200.

Medical diagnosis. A triage decision tree for sepsis risk might ask: heart rate > 90? → temperature > 38.3°C? → suspected infection? → organ dysfunction markers? Each branch is an axis-aligned split on a clinical measurement. This is literally a decision tree — the same recursive binary partitioning, hand-tuned by clinicians. The ML version learns those thresholds from outcomes data.

Credit scoring. A loan approval model uses features like income, debt-to-income ratio, credit utilization, and payment history. At depth 3–5 the tree produces interpretable rules that regulators can audit: "if income < $40k AND debt ratio > 0.45, decline." No black box. This explainability is the primary reason decision trees (and random forests) dominate regulated industries.

What changes in high dimensions. With 200 features, the tree still works — but two problems emerge. First, some features will correlate strongly with the label by chance alone (spurious splits). Second, a single tree becomes brittle: tiny data perturbations change which feature wins the greedy competition at the root, producing completely different trees.

Why random forests dominate single trees in practice: A random forest solves both high-dimensional problems simultaneously. Bootstrap sampling means no single spurious feature appears in all trees. Random feature subsets at each split mean even dominant features are sometimes excluded, forcing the ensemble to find diverse split paths. The result: variance falls dramatically with minimal bias cost. Out-of-bag error — computed on the ~37% of training samples excluded from each tree's bootstrap — gives a free cross-validation estimate without touching the test set.

Common Misconceptions

Five causal mistakes — and the chain that connects them

These aren't five isolated errors. They form a chain — each wrong belief leads directly to the next. Trace the chain and all five become impossible to forget.

  • ① Deeper trees are always better — more splits means more accuracy.
    Only on training data. A fully-grown tree memorizes the training set — every noise artifact gets its own leaf rule. On test data, accuracy collapses. Depth is the primary regularization lever for trees. The optimal depth is almost always far shallower than what unrestricted growth produces.
  • ② The tree keeps overfitting, so it must be choosing the wrong features to split on.
    The split criterion (Gini/entropy) is not the problem — it's correct and well-justified. The problem is that greedy depth-first splitting over-adapts to training data regardless of how good each individual split is. Pruning — not a different criterion — is the fix. Changing to entropy from Gini rarely helps.
  • ③ Pruning fixes overfitting, so a well-pruned single tree should be sufficient.
    Pruning reduces overfitting for a given dataset, but a single decision tree is inherently high variance — small changes in training data can completely restructure it. Which feature wins the greedy competition at the root is sensitive to every sample. A random forest doesn't need a different tree; it needs many diverse trees whose variance cancels.
  • ④ A random forest is just many copies of the same tree trained on the same data.
    Identical trees voting identically adds nothing. A random forest creates diversity two ways: each tree sees a different bootstrap sample (≈63% unique rows), and at each node, only √p features are candidates. These injections of randomness ensure trees make different errors. Averaging diverse errors reduces variance — averaging identical errors does not.
  • ⑤ Since random forests outperform single trees, you should always use them — trees are obsolete.
    Decision trees remain indispensable where interpretability is required: medical triage protocols, credit approval explanations, regulatory audits. A random forest of 500 trees tells you its prediction; a decision tree tells you why in auditable steps. The right tool depends entirely on whether explainability is a constraint. Trees are not obsolete — they're the only model that produces human-readable rules.

Summary

Six things to lock in

  • A decision tree recursively splits the feature space into axis-aligned rectangles, choosing the split that minimizes weighted Gini impurity (or entropy) at each node.
  • Gini impurity = 1 − Σ p_mk². It equals 0 for a pure node, 0.5 for a maximally mixed two-class node. The tree greedily minimizes it — no global optimum guarantee.
  • Depth is the primary regularization knob. Unrestricted growth memorizes training noise. max_depth, min_samples_leaf, and ccp_alpha all constrain this.
  • A single tree is high-variance — tiny training data changes can restructure it completely. This is not fixed by better split criteria; it requires ensemble methods.
  • A random forest decorrelates trees via two mechanisms: bootstrap sampling (different data per tree) and random feature subsets at each split (√p features). Averaging diverse trees cuts variance with minimal bias cost.
  • The interpretability tradeoff is real: a single tree produces auditable rules; a forest of 500 trees does not. Choose based on whether explainability is a requirement, not just performance.
One question this chapter leaves unanswered: Decision trees partition space with axis-aligned cuts — vertical and horizontal lines only. That means a boundary running diagonally through feature space requires an exponentially large number of splits to approximate. Is there a model that can find any linear boundary in a single step — and handle non-linear boundaries just as cleanly by transforming the space?

That's Support Vector Machines. Next: Support Vector Machines →

Test Your Understanding

9 Questions

0 / 9 revealed

01 A node has 10 samples: 7 class A, 3 class B. What is its Gini impurity? ▼
Gini = 0.42. p_A = 7/10 = 0.7, p_B = 3/10 = 0.3. Gini = 1 − (0.7² + 0.3²) = 1 − (0.49 + 0.09) = 1 − 0.58 = 0.42. For comparison: a pure node (10/0 split) has Gini = 0; a 50/50 split gives Gini = 0.5. The algorithm will try to find a split that pushes both child nodes below 0.42.
02 You train a decision tree with no depth limit. Training accuracy is 100%. Test accuracy is 58%. What is happening and what is the simplest fix? ▼
The tree has memorized the training data — this is overfitting. With no depth limit, the tree grows until every leaf contains a single training sample, storing all noise as rules. The simplest fix is setting max_depth to a small value (3–5). Alternatively, min_samples_leaf=5 prevents any leaf from having fewer than 5 examples, forcing the tree to generalize. Cost-complexity pruning (ccp_alpha) is more principled but requires cross-validation to tune.
03 Why does a random forest outperform a well-pruned single decision tree, even when both use the same max_depth? ▼
A single tree is high-variance; an ensemble of diverse trees is not. Even a pruned tree is sensitive to which examples happen to be in the training set — small changes can restructure the tree completely. A random forest trains each tree on a different bootstrap sample (≈63% unique rows) and restricts feature candidates at each split to √p features. This forces trees to make different errors. Averaging diverse errors reduces variance with minimal bias increase — something no amount of single-tree pruning can achieve.
04 What is bagging, and why does it reduce variance without significantly increasing bias? ▼
Bagging = bootstrap aggregating: train many models on random bootstrap samples, then average their predictions. Each bootstrap sample is drawn with replacement from the full training set, so each model sees a different "version" of the data. The key statistical insight: averaging N unbiased estimators with variance σ² and correlation ρ gives variance ρσ² + (1−ρ)σ²/N. As N grows, the second term vanishes. When trees are decorrelated (by feature subsampling), ρ is small and variance drops substantially. Bias is unchanged because each tree is still trained to minimize impurity on a representative sample of the full distribution.
05 You switch a decision tree's criterion from Gini impurity to entropy. Accuracy on the test set barely changes. Why? ▼
Gini and entropy measure similar things and almost always produce identical splits. Gini impurity = 1 − Σp², entropy = −Σp·log(p). Both equal zero for a pure node and are maximized at equal class proportions. They diverge only at extreme distributions, and even then the split selected is usually the same because the greedy algorithm chooses among discrete thresholds. The practical difference is computational: log() is more expensive than squaring. Criterion choice rarely matters; depth and min_samples matter far more.
06 What is out-of-bag (OOB) error in a random forest, and why is it useful? ▼
OOB error is a free cross-validation estimate using samples not seen by each tree. Each bootstrap sample includes ~63% of training rows; the remaining ~37% are "out-of-bag" for that tree. After training, each sample is predicted by only the trees that never saw it during training. The average OOB error across all samples estimates generalization error without requiring a separate validation set. This is especially valuable when data is limited — you get honest generalization estimates without sacrificing training data to a hold-out set.
07 How does a random forest measure feature importance, and what is a known bias in the "mean decrease in impurity" method? ▼
Two methods: permutation importance and mean decrease in impurity (MDI). MDI sums how much each feature reduces weighted impurity across all trees. It's fast but biased toward high-cardinality features — features with many possible split thresholds (like continuous numerical features or IDs) appear more important simply because they offer more splitting opportunities. Permutation importance avoids this: it measures how much test accuracy drops when a feature's values are randomly shuffled. Slower, but unbiased. For reliable feature importance, prefer permutation importance.
08 A decision tree is asked to classify a point near the boundary between two classes. Why might its confidence be unreliable compared to a random forest? ▼
A single tree's decision boundary is a hard axis-aligned cut; it doesn't model uncertainty near boundaries. Near a decision boundary, the leaf covering that region might be 55% class A / 45% class B — a legitimate uncertainty signal. But a single tree's structure depends heavily on which samples fell in which leaf during training. A small perturbation in training data shifts the boundary. A random forest averages probabilities across many trees — each with slightly different boundaries — giving a much more stable estimate of boundary uncertainty. The forest's 55% confidence reflects genuine ambiguity; the single tree's confidence reflects a noisy artifact of one training run.
09 A medical team wants to deploy an ML model to flag high-risk patients. A random forest achieves 92% accuracy; a decision tree achieves 87%. Which should they choose and why? ▼
Likely the decision tree — because interpretability is a requirement in medical deployment. Clinicians need to understand and audit the model's reasoning for every decision. A decision tree produces a set of explicit rules: "if heart rate > 110 AND lactate > 2 → high risk" — which doctors can verify, override, and explain to patients and regulators. A random forest of 500 trees cannot provide this. The 5% accuracy gap must be weighed against the explainability requirement, liability in high-stakes decisions, and regulatory constraints. In practice, teams often start with a decision tree for interpretability, then deploy a forest if regulations permit and the accuracy gap is clinically meaningful.

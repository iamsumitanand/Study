Model Context Protocol (MCP)

The open standard that lets any LLM talk to any tool — without custom glue code.

Before MCP, every AI integration was a custom cable. You wanted your LLM to read a file? Write a function. Query a database? Write another function. Hit a REST API? Yet another function — all one-offs, all incompatible, all duplicated across every team and every product. MCP is the USB port — one protocol, any tool. Build a server once and any MCP-compatible host can use it.

The Intuition

Why a standard protocol matters more than any single implementation

MCP (Model Context Protocol) is an open protocol published by Anthropic in November 2024. Its core insight is architectural: the problem with giving LLMs tools isn't writing the tool — it's the integration tax. Every new tool requires a new adapter, tested against a specific LLM, deployed with a specific runtime. MCP eliminates that tax by defining a universal interface between a host (the application running the LLM), a client (the protocol adapter inside that host), and one or more servers (the tools and data sources).

The architecture separates three concerns that were previously tangled together. The host decides which servers to connect to and how the LLM uses them. The client speaks the protocol. The servers expose capabilities without knowing anything about which LLM is on the other end. This decoupling is the entire value proposition: a filesystem server built today works with Claude, GPT-4, Gemini, and any future model that implements the client side of the spec.

  • The host is the LLM application — Claude Desktop, VS Code with Copilot, a custom agent. It owns the LLM and controls what the user sees.
  • The client lives inside the host and manages exactly one connection to one MCP server. It speaks JSON-RPC 2.0.
  • The server exposes three primitive types: tools (callable functions), resources (readable data), and prompts (reusable prompt templates).
  • The transport layer is pluggable — stdio for local processes, HTTP+SSE for remote servers. The protocol is transport-agnostic.

Understanding It Deeply

Three questions that separate surface understanding from real understanding

MCP looks simple at first: the LLM calls a function, gets a result. But the design decisions — stateful sessions, three capability types, a separate client layer — each solve a specific problem that naive "just call the API" approaches hit in production.

Think first How is MCP different from just calling an API directly inside a function that the LLM is told about? If you can already give an LLM a search_web(query) function, what problem does MCP solve?

Calling an API inside a function works — once, for one LLM, maintained by one team. The difference is standardisation and discoverability. With raw function calling, every integration requires custom code to define the schema, handle errors, manage credentials, and translate results back into something the LLM understands. MCP formalises all of that. An MCP server announces its capabilities at connection time via tools/list, resources/list, and prompts/list. The host discovers what the server can do dynamically — no hardcoded schema, no bespoke adapter. The same server binary can be connected to Claude Desktop, a CI pipeline, and a VS Code extension without modification. It also separates execution from orchestration: the server runs in its own process with its own permissions, so a compromised prompt cannot escape into the host's process space.

Think first MCP exposes three primitive types: resources, tools, and prompts. Before reading further — what do you think distinguishes them? Why not just expose everything as a tool?

The three types exist because they have fundamentally different semantics. Resources are data — files, database rows, live sensor readings. They are identified by URI and read with resources/read. They may change over time (the protocol supports resources/updated notifications), but the LLM doesn't execute them — it reads them. Think of them as the server's file system. Tools are actions — search, create, send, query. They have input schemas and produce output. They can have side effects. The LLM calls them with tools/call, and the server executes the underlying logic. Prompts are reusable, parameterised prompt templates stored server-side. A git server might expose a create_pr_description prompt that takes a diff and returns a structured prompt the host injects into the LLM conversation. This keeps prompt engineering centralised on the server, not scattered across client codebases. Conflating all three into "tools" would lose the read-only safety guarantee of resources and the discoverability of server-curated prompt patterns.

Think first MCP uses a stateful session model: client and server go through an initialisation handshake before exchanging any data. Why does the security model depend on this? What attack becomes possible if you allow stateless, arbitrary requests?

Stateless requests would allow any caller to invoke any capability without establishing trust first. The MCP initialisation handshake serves as a capability negotiation and trust establishment step. During initialize, the client declares its protocol version and capabilities; the server responds with its own version and capability list. This means the server knows exactly which client it is talking to and can enforce capability-specific access controls before any tools are called. More critically, the session model enables consent flows: hosts can prompt users to approve which servers connect and what they are permitted to do, before any tool execution begins. A stateless model would make it trivially easy for a malicious prompt to invoke a server capability without the user ever being aware a connection was established. The session also lets the server maintain per-connection state (e.g. authentication tokens, opened file handles) that would be unsafe to pass in each individual request.

Protocol Deep-Dive

Actual JSON-RPC messages on the wire

MCP uses JSON-RPC 2.0 as its message format. Every request has a jsonrpc field, a numeric or string id, a method, and a params object. Every response echoes the id and carries either a result or an error. Here are the three canonical interactions.

Tool Call — tools/call
REQUEST → tools/call
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "read_file",
    "arguments": {
      "path": "/home/user/notes.txt"
    }
  }
}
RESPONSE ← server
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "Meeting at 3pm. Bring slides."
      }
    ],
    "isError": false
  }
}
Resource Read — resources/read
REQUEST → resources/read
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "resources/read",
  "params": {
    "uri": "file:///etc/schema/orders.sql"
  }
}
RESPONSE ← server
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "contents": [
      {
        "uri": "file:///etc/schema/orders.sql",
        "mimeType": "text/plain",
        "text": "CREATE TABLE orders (id INT, sku VARCHAR, qty INT);"
      }
    ]
  }
}
Prompt Get — prompts/get
REQUEST → prompts/get
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "prompts/get",
  "params": {
    "name": "summarise_pr",
    "arguments": {
      "diff": "@@ -1,3 +1,5 @@ ..."
    }
  }
}
RESPONSE ← server
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "description": "Summarise a pull request from its diff",
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "Summarise this diff in one paragraph:\n@@ -1,3 +1,5 @@ ..."
        }
      }
    ]
  }
}

Notice the consistent shape: every message is JSON-RPC 2.0. The id field pairs requests with responses asynchronously — critical for HTTP+SSE transport where responses may arrive out of order. The content array in tool results uses a typed union ("type": "text", "type": "image", "type": "resource") so clients know how to render or inject each piece without parsing unstructured strings.

From Spec to Production

MCP in the wild — three deployment patterns

MCP shipped with working implementations in November 2024. Within months it had become the de-facto standard for AI tool integration across the developer tooling ecosystem. Here is what real deployments look like.

1 Claude Desktop — filesystem + database access. The reference deployment. Users configure MCP servers in a JSON config file. Claude Desktop spawns each server as a child process (stdio transport) and exposes its tools directly in the conversation. A filesystem server gives Claude read/write access to approved directories; a SQLite server lets it query local databases. The session model means the user grants access once per server, not per query.
2 VS Code / Cursor — IDE-native MCP integration. VS Code's Copilot extension and Cursor both added MCP support in early 2025. MCP servers run alongside the IDE and give the LLM access to the project's git history, test runner, linter output, and documentation. The LLM can call run_tests(), get structured output, and reason about failures — all without the user copy-pasting terminal output. The same server config works in Claude Desktop and in the IDE, because the protocol is host-agnostic.
3 Enterprise connectors — Slack, GitHub, Jira. For remote services, servers run over HTTP+SSE transport and sit behind OAuth. A GitHub MCP server exposes tools like create_issue, list_prs, and merge_branch. A Slack server exposes post_message and list_channels. An enterprise AI assistant can be connected to all three simultaneously — the host manages one client connection per server, and the LLM composes actions across them in a single conversation turn. Critically, each server enforces its own auth and rate limits independently, so connecting Claude to Slack doesn't grant it GitHub access.
The pattern that emerges across all three: MCP moves integration complexity from the model side (the LLM application team) to the tool side (the tool developer). Anthropic, OpenAI, and Google all implementing MCP clients means a tool developer writes one server and immediately reaches every major LLM host. That is the network effect that makes open protocol design so powerful — it's the same reason HTTP beat proprietary web protocols, and USB beat every device-specific connector.

Common Misconceptions

Five causal misconceptions about MCP

These aren't five random mistakes. They follow a causal chain — each one leads naturally to the next. Trace the chain and none of them will stick.

  • ① MCP is just function calling with extra steps.
    Function calling (as in OpenAI's API) is a model capability — the LLM outputs a structured request to invoke a function. MCP is an integration protocol that operates one layer up. It defines how the host discovers available tools, manages server connections, handles authentication, and routes messages — none of which function calling specifies. You still use function calling inside MCP; MCP tells you what functions exist, where they live, and how to talk to them safely across process boundaries.
  • ② MCP requires a cloud connection — it won't work offline or on a local machine.
    The default MCP transport for local servers is stdio: the host spawns the server as a child process and they communicate over standard input/output. No network required. This is how Claude Desktop connects to filesystem and database servers — entirely local, no data leaves the machine. HTTP+SSE is an optional transport for remote servers, not a requirement of the protocol.
  • ③ MCP is Anthropic-specific — it only works with Claude.
    MCP is an open specification (published at modelcontextprotocol.io) and is deliberately model-agnostic. OpenAI, Google, and Microsoft all added MCP client support in 2025. Any application that implements the client side of the JSON-RPC spec can use any MCP server. Anthropic authored the protocol and maintains the reference implementations, but the spec is not Claude-specific — the word "Claude" does not appear in the core protocol spec.
  • ④ The three capability types (tools, resources, prompts) are interchangeable — use whichever is convenient.
    They have distinct semantics that matter for safety and correctness. Resources are read-only data with URI-based addressing — using them signals no side effects. Tools are executable actions that may have side effects — hosts must present appropriate user consent UI before executing them. Prompts are server-curated conversation templates, not data and not executable. Conflating them breaks the permission model: a host that treats a tool as a resource might skip user approval for an action that sends an email or deletes a file.
  • ⑤ MCP solves the context window problem — you can now give an LLM access to unlimited data.
    MCP provides access to data, not unlimited context. The LLM still has a finite context window. MCP shifts the architecture from "load everything upfront" to "fetch on demand" — the LLM calls resources/read to pull in only the data it needs for each step. This is far more efficient than RAG-style pre-loading for structured queries, but the constraint is the same: whatever gets returned by the server must fit in the context. MCP is a retrieval and action mechanism, not a memory system.

Summary

Six things to lock in

  • MCP is an open protocol (JSON-RPC 2.0) published by Anthropic in 2024 for connecting LLM hosts to tools and data sources. It is model-agnostic and transport-agnostic.
  • Three-layer architecture: host (owns the LLM), client (speaks the protocol), server (exposes capabilities). One client per server connection.
  • Three capability primitives: tools (actions, may have side effects), resources (read-only data, URI-addressed), prompts (server-curated prompt templates). They are not interchangeable.
  • Two transports: stdio for local servers (default, no network), HTTP+SSE for remote servers. The protocol is identical over both.
  • Stateful sessions with an initialize handshake: the server announces capabilities, the host can enforce user consent before any tools are called. This is the core of the security model.
  • MCP does not solve the context window — it provides on-demand access, not unlimited memory. Data fetched via MCP still consumes context tokens like any other content.
One question this chapter leaves unanswered: MCP gives an LLM access to tools. But who decides when to call them, in what order, and how to handle a tool result that changes what the next step should be? A single tool call is interesting. A system that dynamically plans and re-plans across many tool calls — perceiving, acting, observing, and reasoning in a loop — is something else entirely.

That's agentic AI. And it's the next chapter.

Test Your Understanding

9 Questions

0 / 9 revealed

01 An MCP host connects to three servers: filesystem, Slack, and GitHub. The LLM calls post_message. Which component decides which server receives the request? ▼
The MCP client (inside the host). The host maintains one MCP client per server connection. When the LLM generates a tool_use block, the host identifies which server advertised post_message during its tools/list response, then routes the tools/call request through the corresponding client connection. Servers have no visibility into each other; the host is the router.
02 What is the difference between an MCP resource and an MCP tool? Give a concrete example where the distinction matters for user safety. ▼
Resources are read-only data; tools are executable actions that may have side effects. A resource is addressed by URI and read with resources/read — it cannot modify state. A tool is invoked with tools/call and can do anything: write files, send messages, delete records. The distinction matters for consent: a host can safely inject a resource's content into the LLM context without user approval (it's just reading data). But executing a tool — especially one that sends an email or posts to Slack — requires presenting the user with an approval prompt. Conflating the two would mean silently performing irreversible actions that the user never knowingly authorised.
03 A developer wants to run MCP on an air-gapped machine with no network access. Is this possible? What transport should they use? ▼
Yes — use the stdio transport. MCP's stdio transport requires no network. The host spawns the MCP server as a child process and communicates over standard input/output. The JSON-RPC messages are identical to the HTTP transport; only the delivery mechanism differs. This is the default for Claude Desktop's local server integrations and is the recommended transport for any server running on the same machine as the host.
04 During MCP initialisation, the client sends an initialize request. What two things does the server return in its response, and why do both matter? ▼
Protocol version and server capabilities. The protocol version matters because MCP is versioned — a server that supports protocol 2025-03 may expose features unavailable in 2024-11. The client and server negotiate a shared version they both support, preventing one side from sending messages the other cannot parse. The capabilities object lists which feature categories the server supports (e.g. tools, resources, prompts, logging) so the client knows what to request via list calls — avoiding wasted round-trips to servers that don't implement a given feature.
05 MCP uses JSON-RPC 2.0. Every request includes an id field. Why is this field important, and what would break without it? ▼
The id field pairs asynchronous responses to their requests. Over HTTP+SSE (and even stdio with pipelined requests), the host may send multiple requests before receiving any responses. Without an id, the client cannot know which response corresponds to which request. For example, if the LLM calls read_file and run_query simultaneously, both responses arrive as separate JSON objects with no implicit ordering. The matching id is the only mechanism connecting them. Without it, concurrent tool calls would be impossible, forcing strictly sequential execution and eliminating the performance benefit of parallel server queries.
06 A developer builds one MCP server for their internal wiki. They want it to work with Claude Desktop, a VS Code extension, and a custom Python agent. Do they need to write three different servers? ▼
No — one server, multiple hosts. MCP's model-agnostic design means any host that implements the client-side protocol can connect to any compliant server. Claude Desktop, VS Code Copilot, and a Python agent using the MCP SDK all speak the same JSON-RPC protocol. The server exposes its capabilities once via tools/list and resources/list, and each host connects independently. This is the core value proposition: the integration is written once on the server side and reused across all host implementations.
07 MCP's tool call response includes an isError boolean. Why is this better than using the JSON-RPC error field for tool failures? ▼
Because isError lets the LLM see and reason about the failure as part of its context. JSON-RPC's error field signals a protocol-level failure — the call couldn't be made at all (bad method, invalid params, network error). isError: true inside a successful JSON-RPC response means the tool ran but the underlying operation failed — e.g. "file not found" or "permission denied". The error message is returned as content, so it lands in the LLM's context window. The LLM can then reason about it: "the file doesn't exist, I should try a different path" rather than crashing. Using the JSON-RPC error channel for tool-level failures would prevent the LLM from recovering gracefully.
08 An enterprise connects their AI assistant to GitHub via MCP. A malicious prompt injection in a GitHub issue tries to make the assistant call delete_repository. What MCP mechanism is the first line of defence? ▼
User consent enforcement in the host before tool execution. The MCP spec requires hosts to present users with approval before executing tools that have side effects. Even if the LLM generates a tools/call for delete_repository, the host should surface a prompt to the user: "The assistant wants to delete repository X. Allow?" The user can deny it. Beyond consent, servers can also enforce capability-level restrictions during initialisation — a read-only GitHub integration simply doesn't advertise destructive tools in its tools/list response, so the host never presents them to the LLM in the first place.
09 MCP gives the LLM access to a large internal knowledge base via resources/read. A user asks a question whose answer spans 500 documents. Does MCP solve this? Why or why not? ▼
Not directly — MCP provides access, not synthesis across unlimited context. The LLM can call resources/read on documents one by one, but each response consumes context tokens. Fetching all 500 documents would overflow the context window. What MCP enables is targeted retrieval: the LLM can call a tool that searches the knowledge base (e.g. a semantic search tool returning the top 5 most relevant chunks), then read only those. This is architecturally similar to RAG but driven by the LLM's own queries rather than a fixed pre-retrieval step. For true multi-document synthesis, you still need either very long context models, iterative summarisation loops, or purpose-built RAG pipelines. MCP is the transport for those patterns, not a replacement for them.

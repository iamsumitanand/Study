Part 1 · Basic Machine Learning

Evaluation Metrics

Measuring what your model actually does — not what you hope it does.

Accuracy: 99% on cancer screening. The model predicts "no cancer" for everyone. Congratulations? If 99% of patients are healthy, you just built the most useless model imaginable — and your metric rewarded you for it. Every number we use to judge a model has a hidden assumption baked in. The skill isn't computing metrics; it's knowing which ones lie to you and when.

The Intuition

The right metric for the right problem

Accuracy is the ratio of correct predictions to total predictions. In balanced datasets, it's perfectly reasonable. In imbalanced ones — where one class vastly outnumbers the other — it actively hides failure. A model that never fires a positive prediction can achieve 99% accuracy on a 1% positive-rate dataset while being completely worthless.

The root problem is class imbalance: real-world classification tasks are rarely 50/50. Cancer is rare. Fraud is rare. Equipment failure is rare. When the negative class dominates, accuracy measures how well your model ignores the positives — which is precisely backwards from what you care about.

Different tasks demand different trade-offs. Cancer screening demands you catch every case, even at the cost of false alarms. Spam filtering demands you don't bury real emails, even at the cost of some spam slipping through. Fraud detection demands you don't flag innocent customers, even at the cost of missing some fraud. No single metric captures all of these simultaneously — which is why the field developed Precision, Recall, F1, and AUC: each one answers a different question about what "good" means.

  • Precision — of everything I labeled positive, how many actually were? (False alarm rate)
  • Recall — of everything that actually was positive, how many did I find? (Miss rate)
  • F1 — a single number balancing both when neither can be sacrificed entirely
  • AUC — how well can the model separate classes across all possible thresholds?

Understanding It Deeply

The trade-offs that can't be avoided

Every binary classifier assigns a probability score, then makes a decision at a threshold. Shift the threshold and you fundamentally change what the model values. Understanding metrics means understanding what you're implicitly optimizing when you choose one.

Think first Maximizing Recall means catching more true positives — but what happens to false positives as you lower the threshold to catch more cases? What does that mean in a medical screening scenario for both the patient and the healthcare system?

Maximizing Recall pushes the threshold down: you flag more cases as positive, catching more true positives — but also generating more false positives. In medical screening, this means more unnecessary follow-up tests, procedures, and patient anxiety for people who are actually healthy. The healthcare system bears the cost of those false alarms. This is a deliberate trade: you accept more false positives to ensure you miss fewer true positives. Whether that trade is worth it depends entirely on the relative severity of missing a case versus triggering a false alarm. For late-stage cancer, missing a case is catastrophic; a biopsy for a healthy person is stressful but manageable. So high Recall is the right choice — but the cost is real.

Think first AUC measures performance across all thresholds simultaneously. Can you imagine a situation where a model with high AUC would still be useless in practice — or even dangerous?

AUC is misleading when the class imbalance is extreme or the cost asymmetry is large. A model can achieve 0.95 AUC while performing terribly at the specific operating threshold you actually deploy it at. AUC also ignores calibration: a model that ranks positives above negatives consistently but with poorly calibrated probabilities will have high AUC but fail when you try to make actual probability-based decisions. In fraud detection, a high-AUC model that also fires false positives on high-value customers may cost more in customer churn than the fraud it catches.

Think first F1 is a single number. So is accuracy. So is AUC. If every metric eventually collapses to a scalar, why can't any single number fully capture model quality?

Because model quality is inherently multidimensional. Any single scalar aggregates across conditions and compresses information. F1 treats false positives and false negatives as equally costly — which is almost never true in practice. AUC averages over thresholds that you will never deploy. Accuracy aggregates across classes with different importances. The right answer is: know what decision your model ultimately feeds into, and optimize the metric that reflects the cost structure of that decision. No pre-packaged single number will do that for you automatically.

The Math

Four formulas and what they measure

Everything follows from the four cells of the confusion matrix: true positives (TP), false positives (FP), true negatives (TN), and false negatives (FN). Each metric is a ratio that emphasizes a different combination.

Precision
Precision = TP / (TP + FP)
TP true positives — predicted positive, actually positive FP false positives — predicted positive, actually negative (false alarms)
Recall (Sensitivity)
Recall = TP / (TP + FN)
TP true positives — cases correctly found FN false negatives — cases missed (the model said "no" when it should have said "yes")
F1 Score
F1 = 2 · P · R / (P + R)
P Precision — harmonic mean penalizes extreme imbalance between P and R R Recall — F1 is low whenever either P or R is low, even if the other is perfect
AUC (Area Under the ROC Curve)
AUC = ∫₀¹ TPR(FPR) d(FPR)
TPR true positive rate = Recall — fraction of positives correctly found at a given threshold FPR false positive rate = FP/(FP+TN) — fraction of negatives incorrectly flagged AUC probability that a randomly chosen positive ranks higher than a randomly chosen negative

From Toy to Real

Three tasks, three right metrics

The same model optimized with different objectives becomes a different tool. These three domains show why the choice of metric is an engineering decision with real-world consequences — not a statistical formality.

Medical Cancer screening — maximize Recall

A false negative (missed cancer) leads to delayed treatment, potentially fatal. A false positive (unnecessary biopsy) is stressful and costly, but manageable. The cost asymmetry is enormous. Optimize for Recall above 0.95 — accept lower Precision as the price. The threshold gets set very low: "when in doubt, flag it." Accuracy would reward the lazy model that never flags anything.

Email Spam detection — balanced F1

A false negative (spam in inbox) is annoying. A false positive (real email in spam) is often worse — a user misses a job offer or invoice. Neither failure mode dominates clearly. F1 is the right target: it punishes models that sacrifice either Precision or Recall too heavily. Modern spam filters tune F1 on large labeled datasets and accept that neither error rate will reach zero.

Fraud Fraud detection — maximize Precision

A false positive (blocking a legitimate transaction) damages customer trust, triggers support calls, and may end the banking relationship. False negatives (missed fraud) are absorbed by insurance and fraud departments — painful, but diffuse. High-Precision models fire only when confident. The threshold is set high. Recall suffers — some fraud slips through — but customers are never wrongly blocked on high-confidence legitimate transactions.

Common Misconceptions

Five wrong beliefs — and how they connect

These aren't five random mistakes. Each one is the natural consequence of the one before it. Trace the chain and none of them can survive.

  • ① "Accuracy is a reliable model evaluation metric."
    Only on balanced datasets. Accuracy counts all errors equally regardless of class. On a dataset that is 99% negative, a model that predicts "negative" for everything achieves 99% accuracy while detecting zero positives. Accuracy is the right metric only after you've confirmed your dataset is roughly balanced — which is rare in practice. This is the root error; every misconception downstream inherits it.
  • ② "If accuracy misleads, just look at both classes separately — accuracy ignores imbalance, but that doesn't affect Precision or Recall."
    Imbalance affects everything. Precision is affected because having many negatives inflates the denominator of FP-based calculations when a model fires carelessly. Recall is affected because a model that learns to say "positive" rarely (to avoid FP) will have low Recall regardless of how balanced you think the framing is. Imbalance affects which errors occur — not just how you count them.
  • ③ "Fine — but Precision and Recall don't matter until deployment. During training, just optimize the loss."
    Cross-entropy loss during training optimizes predicted probabilities, not decisions. At deployment, you apply a threshold to turn probabilities into decisions — and Precision/Recall are properties of that decision boundary, not the probabilities. A model with excellent cross-entropy loss can produce terrible Precision or Recall depending on where you set the threshold. The two are genuinely separate concerns.
  • ④ "Then just tune the threshold to hit the Precision you want — one metric is enough."
    Moving the threshold to improve one metric always degrades another. Raising it improves Precision but hurts Recall. Lowering it does the reverse. You cannot optimize both simultaneously with a single threshold — this is the Precision-Recall trade-off, and it's a mathematical constraint, not a tuning problem. F1 helps by enforcing a balance, but any single scalar still hides the trade-off it made.
  • ⑤ "AUC = 0.97 means the model is excellent — that's near-perfect."
    AUC measures rank separation across all thresholds equally — including thresholds you will never deploy at. It says nothing about performance at the threshold you actually use. A model with 0.97 AUC can have mediocre Precision or Recall at your operating threshold. It also ignores calibration: if the probability scores are poorly calibrated, downstream decisions using those probabilities will be wrong even when AUC looks great. AUC is a useful screening tool, not a final verdict.

Summary

Five things to lock in

  • Accuracy is unreliable on imbalanced datasets — it rewards models that ignore the minority class entirely. Always check class distribution before reporting accuracy.
  • Precision = TP/(TP+FP) measures false alarm rate. Recall = TP/(TP+FN) measures miss rate. They trade off against each other as you move the threshold.
  • F1 is the harmonic mean of Precision and Recall — it is low whenever either is low, even if the other is perfect. Use it when neither failure mode can be ignored.
  • AUC measures how well the model ranks positives above negatives across all thresholds. Useful for comparing models, but tells you nothing about performance at any specific threshold.
  • The right metric is determined by the cost asymmetry of the deployment decision — medical screening wants high Recall, fraud detection wants high Precision, spam filtering wants balanced F1.

Test Your Understanding

9 Questions

0 / 9 revealed

01 A model achieves 99% accuracy on a medical test. The disease affects 1% of the population. The model always predicts "no disease." Is this a useful model? ▼
No — it detects nothing. Recall = 0 because every true positive is missed (TP=0, FN=all positives). Precision is undefined (no positive predictions). F1 = 0. The 99% accuracy comes entirely from correctly predicting negatives for the 99% healthy population. This is the canonical class-imbalance failure mode: accuracy rewards the silent model.
02 TP=40, FP=10, TN=30, FN=20. Compute Precision, Recall, and F1. ▼
Precision = 0.80, Recall = 0.67, F1 = 0.727. Precision = 40/(40+10) = 40/50 = 0.80. Recall = 40/(40+20) = 40/60 = 0.667. F1 = 2·0.80·0.667/(0.80+0.667) = 1.067/1.467 = 0.727. Accuracy = (40+30)/100 = 0.70. Note F1 (0.727) is higher than accuracy (0.70) here because the model performs better on positives than negatives.
03 You lower the classification threshold from 0.5 to 0.2. What happens to Precision and Recall? Why? ▼
Recall increases; Precision decreases. Lowering the threshold means more samples are classified as positive. You catch more true positives (Recall goes up) but also more false positives (Precision goes down — the model is now less selective). This is the fundamental Precision-Recall trade-off: you cannot improve both simultaneously by moving a single threshold.
04 Model A has AUC=0.95. Model B has AUC=0.80. Does this mean Model A is better for a fraud detection deployment? ▼
Not necessarily. AUC averages performance across all thresholds. Fraud detection deploys at a specific threshold (typically high Precision to avoid blocking customers). Model A could have better overall ranking but worse Precision at the deployment threshold than Model B. Also, AUC ignores calibration: if Model A's probability scores are poorly calibrated, the chosen threshold will not correspond to its intended precision. Always evaluate at the actual operating threshold.
05 What does a point on the ROC curve represent? ▼
A specific (FPR, TPR) pair at one threshold. For a given threshold value, FPR = FP/(FP+TN) — how many negatives are incorrectly flagged — and TPR = TP/(TP+FN) = Recall — how many positives are correctly found. The ROC curve traces all such pairs as the threshold sweeps from 1 (predict nothing) to 0 (predict everything). The top-left corner (FPR=0, TPR=1) is perfect. The diagonal is a random classifier.
06 Why is F1 a harmonic mean rather than an arithmetic mean of Precision and Recall? ▼
The harmonic mean penalizes extreme imbalance between the two values. Arithmetic mean of 0.9 and 0.1 = 0.5 — looks mediocre. Harmonic mean = 2·0.9·0.1/(0.9+0.1) = 0.18 — correctly signals that one metric is terrible. The harmonic mean is dominated by whichever value is lowest, which matches the intuition: a model that is great at Precision but has 10% Recall is not a "0.5-quality" model — it's failing badly at finding positives.
07 A random classifier (coin flip) achieves what AUC? Why? ▼
AUC = 0.5. A random classifier assigns scores uniformly at random. The probability that a randomly chosen positive scores higher than a randomly chosen negative is exactly 0.5 — pure chance. On the ROC curve, this produces the diagonal line from (0,0) to (1,1). The area under the diagonal is 0.5. Any useful model should have AUC > 0.5; AUC < 0.5 means the model is systematically inverting its predictions (which you can fix by flipping the output).
08 Two spam filters have the same F1. Filter A has Precision=0.99, Recall=0.51. Filter B has Precision=0.72, Recall=0.72. Which would you deploy, and why? ▼
Filter B for most email use cases. Filter A has very high Precision — almost no false positives (real email goes to spam rarely) — but lets through nearly half of all spam. Filter B blocks spam more aggressively and misfiles real email more often, but at a manageable rate. For personal email where missing real mail is costly, Filter A is safer. For enterprise bulk inboxes where half your spam leaking through is unacceptable, Filter B wins. This illustrates why F1 alone doesn't decide — you need to know the deployment context and cost structure.
09 Can Precision be 1.0 and Recall be 0.0 simultaneously? What model behavior would produce this? ▼
Yes — a model that makes exactly one positive prediction, and gets it right. If TP=1, FP=0, FN = all remaining positives: Precision = 1/(1+0) = 1.0 and Recall = 1/(1 + many FN) ≈ 0. More generally, a model that fires very rarely but always correctly gets Precision=1.0 while having very low Recall. This is why optimizing Precision alone can be gamed by a model that almost never predicts positive — it achieves perfect Precision while catching essentially nothing.
